# GKGuard v0.1.37

## 变更摘要

`v0.1.37` 是 `v0.1.36` 安全加固后的体验与性能跟进版本，重点优化结果页左侧检索记录切换时的关键帧加载。GKGuard C2 前端会保留当前关键帧，预加载并解码目标记录关键帧，加载完成后再替换详情图；快速切换记录时会让旧加载任务失效，关键帧加载失败后允许后续点击重新请求。GKGuard C2 后端对成功读取的 CampusVision C1 媒体响应使用按响应地址、API key、候选配置和连接代次隔离的进程内短期缓存，并复用短期 CampusVision C1 健康状态探测结果；媒体请求发起前、返回前以及旧连接返回错误前会重新校验连接代次，变化时重新解析候选或返回可重试错误。结果页中等宽度布局会约束人物照片列宽，避免人物照片遮挡数据来源和命中记录信息。

## 主要更新

- 结果页切换左侧检索记录时，不再先清空右侧关键帧区域；当前关键帧会保持可见，目标关键帧加载完成后再替换。
- 快速切回已显示记录时会取消旧关键帧加载任务，避免旧 preload 完成后覆盖当前记录的标题、详情和人脸框。
- 关键帧预加载失败会清理对应 URL 缓存，后续重新选择同一记录会再次请求媒体资源。
- 关键帧加载期间显示轻量提示，避免 CampusVision C1 媒体响应较慢时出现整块黑屏。
- 结果列表切换只同步激活状态，不再重建整组记录卡片，减少缩略图重载和布局抖动。
- GKGuard C2 `/c1/media/{kind}/{face_id}` 对成功媒体响应增加进程内短期缓存，缓存键包含实际响应的 CampusVision C1 base URL、API key、候选地址配置和连接代次；默认 300 秒、最多 64 项，总上限 16 MiB、单项上限 2 MiB，可通过 `C1_MEDIA_CACHE_TTL_SEC`、`C1_MEDIA_CACHE_MAX_ITEMS`、`C1_MEDIA_CACHE_MAX_BYTES` 和 `C1_MEDIA_CACHE_MAX_ITEM_BYTES` 调整。
- 已缓存且未过期的关键帧或人脸裁剪图会先于实时 CampusVision C1 状态探测返回，降低服务短暂不可达时的记录切换失败率；`GET /c1/status` 或桌面端 SSH 隧道重新确认会推进连接代次并清理旧媒体缓存。
- 媒体缓存写入绑定实际响应的 CampusVision C1 地址，避免并发请求切换候选地址时把一个 CampusVision C1 实例的媒体写到另一个实例的缓存键下。
- 若媒体请求已解析出候选地址，但在健康探测或实际发起媒体 GET 前连接代次发生变化，GKGuard C2 会重新解析候选或返回可重试错误，避免向旧 CampusVision C1 实例发出新的媒体请求。
- 若媒体请求发出后连接代次已被其他流程推进，GKGuard C2 会丢弃旧媒体响应或旧媒体错误并重新解析候选或返回 409 可重试错误，不会把旧 CampusVision C1 实例的媒体或错误直接返回给前端、写入新代次缓存或回写选中地址。
- 同一次媒体请求内若已选候选失效并回退到可用候选，GKGuard C2 会保留成功候选并写入对应代次缓存，避免后续记录切换反复请求失效候选。
- GKGuard C2 普通 CampusVision C1 代理请求复用短期健康状态探测结果，默认 15 秒，可通过 `C1_STATUS_CACHE_TTL_SEC` 调整；`GET /c1/status` 仍执行实时探测，探测失败或身份校验失败会清理对应地址的旧健康缓存。
- 中等宽度结果页会约束人物照片列宽，避免人物照片覆盖数据来源和命中记录信息。
- 协作流程文档补充 PR 正文 reaction 合并门槛：审查状态 reaction 只存在且只读取 PR 正文下方的 reaction，不存在于 review 后评论；PR 正文 👀 表示仍在 review，PR 正文 👍 或明确 approval 表示当前无阻断问题。
- 静态资源版本参数更新为 `v0.1.37-ui`，避免安装版继续复用旧前端资源。

## 行为边界

- 本版不改变 CampusVision C1 检索算法、相似度计算、结果排序或 API 字段语义。
- 媒体缓存仅保存在 GKGuard C2 进程内，不写入仓库或持久化数据目录；进程重启后缓存清空。
- 媒体缓存只用于成功读取的关键帧和人脸裁剪图，并按 CampusVision C1 响应地址、API key、候选配置和连接代次隔离；CampusVision C1 无匹配、超时、检测失败和真实检索失败仍按既有错误恢复流程处理。

## 安装与更新

- Windows：下载并运行 `GKGuard-Setup-0.1.37.exe`。已安装旧版时，可使用软件右上角 `检查更新` 入口更新。
- macOS：下载 `GKGuard-macOS-*.dmg` 或 `GKGuard-macOS-*.zip`。当前仍是未签名、未公证的内部测试包。
- Linux：下载 `GKGuard-Linux-*.AppImage` 或 `GKGuard-Linux-*.deb`。
- Release 中应同时包含 `.blockmap`、`latest.yml`、`latest-mac.yml` 和 `latest-linux.yml`。

# GKGuard v0.1.37

## Summary

`v0.1.37` is a usability and performance follow-up after the `v0.1.36` security hardening. It focuses on keyframe loading when users switch result records from the left list. The GKGuard C2 frontend keeps the current keyframe visible, preloads and decodes the target record keyframe, and replaces the detail image only after loading finishes; fast record switching invalidates stale load tasks, and failed keyframe preloads can be retried on later clicks. The GKGuard C2 backend also uses a short-lived in-process cache for successful CampusVision C1 media responses scoped by responding base URL, API key, candidate configuration, and connection generation, and reuses short-lived healthy CampusVision C1 status probes. Before sending media requests, before returning media responses, and before surfacing errors from an old connection, it re-checks the connection generation and re-resolves candidates or returns a retryable error if the generation changes. Medium-width result layouts constrain the target-portrait column so the portrait cannot cover the source and hit-count summary.

## Highlights

- Switching result records no longer clears the right-side keyframe area first; the current keyframe remains visible until the target keyframe is ready.
- Switching quickly back to the currently displayed record cancels stale keyframe loads, so an old preload cannot overwrite the current title, details, or face box.
- Failed keyframe preloads evict the failed URL from cache, so selecting the same record later retries the media request.
- A lightweight loading hint appears while the target keyframe is loading, so slow CampusVision C1 media responses do not create a full black detail panel.
- Result-list switching updates only the active state instead of rebuilding all record cards, reducing thumbnail reloads and layout churn.
- GKGuard C2 `/c1/media/{kind}/{face_id}` now uses a short-lived in-process cache for successful media responses. The cache key includes the responding CampusVision C1 base URL, API key, candidate-URL configuration, and connection generation; it defaults to 300 seconds, 64 items, a 16 MiB total limit, and a 2 MiB per-item limit. Tune it with `C1_MEDIA_CACHE_TTL_SEC`, `C1_MEDIA_CACHE_MAX_ITEMS`, `C1_MEDIA_CACHE_MAX_BYTES`, and `C1_MEDIA_CACHE_MAX_ITEM_BYTES`.
- Valid cached keyframes or face crops are returned before a fresh CampusVision C1 status probe, reducing record-switch failures when the service is briefly unavailable; `GET /c1/status` or a desktop SSH-tunnel reconfirmation advances the connection generation and clears old media cache entries.
- Media cache writes are bound to the responding CampusVision C1 address, preventing concurrent candidate switches from storing media from one CampusVision C1 instance under another instance's cache key.
- If a media request has resolved a candidate address but the connection generation changes during healthy-candidate probing or before the media GET is sent, GKGuard C2 re-resolves the candidate or returns a retryable error instead of sending a new media request to the old CampusVision C1 instance.
- If another flow advances the connection generation after a media request is sent, GKGuard C2 discards the stale media response or stale media error and re-resolves candidates or returns a retryable 409 error instead of returning media or media errors from the old CampusVision C1 instance to the UI, writing them into the new-generation cache, or overwriting the selected address.
- If the selected candidate fails and the same media request falls back to a healthy candidate, GKGuard C2 preserves the successful candidate and caches the response under that resulting generation, avoiding repeated requests to the failed candidate during later record switches.
- Normal GKGuard C2 CampusVision C1 proxy requests reuse short-lived healthy status probes, defaulting to 15 seconds through `C1_STATUS_CACHE_TTL_SEC`; `GET /c1/status` still performs a fresh probe and clears stale healthy cache entries when a probe or identity check fails.
- Medium-width result layouts now constrain the target-portrait column so the portrait cannot cover the source and hit-count summary.
- Collaboration-flow docs now include the PR-body reaction merge gate: review-status reactions exist only below the PR body, not below review comments; PR-body 👀 means review is still in progress, while PR-body 👍 or explicit approval means there is no current blocker.
- The static asset parameter is updated to `v0.1.37-ui` so installed apps do not keep using stale frontend assets.

## Behavior Boundary

- This release does not change the CampusVision C1 matching algorithm, similarity calculation, result ordering, or API field semantics.
- Media cache is kept only inside the GKGuard C2 process. It is not written to the repository or a persistent data directory, and it is cleared when the process restarts.
- The media cache applies only to successfully fetched keyframes and face crops, and is scoped by CampusVision C1 responding address, API key, candidate configuration, and connection generation. CampusVision C1 no-match, timeout, detection-failure, and real-search-failure states continue to use the existing recovery behavior.

- Linux: download `GKGuard-Linux-*.AppImage` or `GKGuard-Linux-*.deb`.
- The Release should also include `.blockmap`, `latest.yml`, `latest-mac.yml`, and `latest-linux.yml`.
